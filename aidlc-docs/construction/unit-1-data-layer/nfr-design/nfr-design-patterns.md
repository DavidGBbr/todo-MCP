# NFR Design Patterns — Unit 1: Data Layer

## Resilience Patterns

### Pattern: Context-Manager Session Lifecycle
Every DB operation runs inside the `get_session()` async context manager. On success it commits; on any exception it rolls back and re-raises. Resources (connections) are always returned to the pool — no connection leaks on error paths. This directly satisfies SECURITY-15 (fail-closed, resource cleanup).

```python
# Canonical usage — every repository call is wrapped this way
async with get_session() as session:
    repo = TodoRepository(session)
    result = await repo.create(data)
# commit on exit, rollback on exception — automatic
```

### Pattern: Pool Pre-Ping
`pool_pre_ping=True` is set on the async engine. Before each connection is used from the pool, SQLAlchemy issues a lightweight `SELECT 1` to verify liveness. Stale connections (e.g. after Postgres restart) are silently replaced rather than surfacing as errors. No retry logic needed in application code.

### Pattern: Fail-Closed Validation
All inputs are validated by Pydantic v2 before any DB operation is attempted. If validation fails, the repository is never called — the operation fails cleanly with a `VALIDATION_ERROR`. This ensures the DB is never in contact with malformed data (SECURITY-05, SECURITY-15).

## Performance Patterns

### Pattern: Separate COUNT + SELECT for Pagination
`list_todos` issues two queries:
1. `SELECT COUNT(*) FROM todos WHERE <filters>` — gets total without loading rows
2. `SELECT * FROM todos WHERE <filters> ORDER BY <sort> LIMIT <n> OFFSET <m>` — loads only the requested page

This avoids loading the full result set into memory to count it.

### Pattern: Index-Supported Filtering
The following indexes are declared in the Alembic migration to support common filter combinations without full-table scans:

| Index | Columns | Supports |
|---|---|---|
| `ix_todos_deleted_at` | `deleted_at` | Default active-only filter |
| `ix_todos_status` | `status` | Status filter |
| `ix_todos_priority` | `priority` | Priority filter + sort |
| `ix_todos_due_date` | `due_date` | Due-date range filters + sort |
| `ix_todos_project` | `project` | Project exact-match filter |
| `ix_todos_assignee` | `assignee` | Assignee exact-match filter |

Tags use a GIN index on the `ARRAY` column to support the `&&` (any-of) operator efficiently.

### Pattern: Eager Subtask Loading
Subtasks are always loaded with their parent Todo in the same query (SQLAlchemy `selectinload`). No lazy loading — avoids N+1 queries when listing todos with subtasks.

## Security Patterns

### Pattern: Parameterised Queries Only (SECURITY-05)
All SQL is generated by SQLAlchemy's query builder. No string concatenation with user input is used anywhere. User-supplied filter values (status, tags, project, assignee, dates) are passed as bound parameters. This prevents SQL injection by construction.

### Pattern: Credential Isolation (SECURITY-09)
`DATABASE_URL` is the only credential Unit 1 handles. It is:
- Read exclusively from the `DATABASE_URL` environment variable via `pydantic-settings`
- Never logged (repository and session code must not log connection strings)
- Never hardcoded in source or test fixtures (tests use a separate `TEST_DATABASE_URL` env var or testcontainers-generated URL)

### Pattern: Safe Deserialisation (SECURITY-13)
All external data entering the system (MCP tool arguments) is deserialised through Pydantic v2 models with explicit field types and constraints. No `pickle`, no `eval`, no `json.loads` on raw user strings without a schema.

### Pattern: Dependency Pinning (SECURITY-10)
`uv.lock` is committed and used in Docker builds via `uv sync --frozen`. No dependency can silently upgrade between builds. A `pip-audit` or `uv audit` step in CI scans for known CVEs against the lock file.

## Maintainability Patterns

### Pattern: Repository Abstraction
All SQL is encapsulated in `TodoRepository`. The service layer (`TodoService` in Unit 2) only calls repository methods — it never writes SQL directly. This makes the data layer independently testable and replaceable.

### Pattern: Alembic Revision Discipline
- Each schema change (new column, new index, new table) lives in its own numbered revision
- `alembic upgrade head` is idempotent — running it twice is safe
- Downgrade scripts are provided for every revision
- No `autogenerate` in CI — revisions are reviewed before merge

### Pattern: Hypothesis Generator Reuse (PBT-07)
Domain-specific Hypothesis strategies (e.g. `todo_strategy()`, `create_todo_input_strategy()`) are defined once in `tests/conftest.py` or a shared `tests/strategies.py` module and imported by all PBT test files. Generators respect all field constraints (max lengths, valid enums, valid date ranges).
